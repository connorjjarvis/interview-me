# interview-me-senior Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build `interview-me-senior`, a standalone, GitHub-distributable Claude Code skill that runs a staff-level engineering interview on a completed piece of work, grilling the user on their decisions and scoring them against a configurable rubric.

**Architecture:** The skill is authored as runtime instructions (`SKILL.md`) that Claude follows using its own tools (git, file read, browser). Deterministic assets ship alongside: a token-based HTML report template the skill string-substitutes, a sample scorecard that doubles as the output contract, a README, and an MIT LICENSE. There is no compiled code and no external runtime dependency.

**Tech Stack:** Markdown (skill + docs), YAML frontmatter, self-contained HTML/CSS/SVG (no JS framework, no CDN). Git for distribution.

## Global Constraints

- **Skill format:** `SKILL.md` MUST have valid YAML frontmatter with `name: interview-me-senior` and a `description:` that triggers on "interview me", "grill me on this", "review my decisions", "score my work".
- **Stack-agnostic:** No hardcoded Azure DevOps / Jira / GitHub-issue integration. Work-item context is user-pasted; environment tooling is used only if already present.
- **Self-contained HTML:** report template has NO external requests — no CDN scripts, no external stylesheets, no remote fonts/images. Inline CSS + inline SVG only.
- **Artifact safety (spec §6a) are HARD requirements:** out-of-repo save default (`~/.interview-me/<repo-name>/`); in-repo overrides must be gitignored + tracked-check warned; `include_transcript` default on, `include_diff_snippets` default off, `redact_secrets` default on, `auto_open` default off.
- **Cross-platform:** paths must work on Windows, macOS, Linux. `~` expands to the user home dir on all three.
- **License:** MIT.
- **No collaboration drift:** the skill interrogates and scores; it MUST NOT invoke `brainstorming` or help improve the code.
- **Spec is source of truth:** `docs/2026-07-07-interview-me-senior-design.md`. Section references (§) below point into it.

- Produces: a token-based HTML file. The skill (Task 5) substitutes these tokens at report time:
  - `{{TITLE}}` — e.g. "Interview — feature/retry-logic — 2026-07-07"

### Task 5: SKILL.md — five-phase runtime logic

**Files:**
- Modify: `interview-me-senior/SKILL.md` (append the phase instructions after the config block)

**Interfaces:**
- Consumes: config keys (Task 4), the HTML template + its tokens (Task 2), the scorecard shape (Task 3).
- Produces: the complete runtime behavior. This is the core of the skill.

Write each phase as explicit, imperative instructions to the executing agent. Cover, in order:

- [ ] **Step 1: Phase 1 — Recon (silent)**

Instructions: determine the diff range (current branch vs its merge-base with the default branch; if none, staged changes; else last commit) and read it. If `work_item` is in `context_sources`, ask the user to paste a ticket URL/ID or its text — and note that if ADO/Jira/GitHub-issue tooling is already available in the environment it may be used to fetch parents, otherwise use only what's pasted (no hardcoded system). Ask the user for a one-line "what were you trying to do." Do NOT ask interview questions yet. Explicit rule: do not invoke the brainstorming skill or offer to improve the code.

- [ ] **Step 2: Phase 1b — Build the decision map**

Instructions per spec §4: classify the diff into decision points, each tagged with {what you did, 2-3 plausible alternatives a senior would weigh, the trade-off axis it tests}. Rank by "how much judgment does this reveal." Select the top `core_questions` points.

- [ ] **Step 3: Phase 2 — Calibrate**

Instructions: tell the user the review is ready and state the bar (`bar_level`). Wait for the user to confirm they're ready before the first question.

- [ ] **Step 4: Phase 3 — Interview (one question at a time)**

Instructions: ask one grounded question per decision point ("you did X, why not Y?"). Apply `tone`. After the answer, score it privately against the relevant dimension(s) using the §5 rubric anchors. If the answer scores low/vague, ask exactly one follow-up probe (respecting `probe_depth`) before moving on; a good probe answer can raise the score. One question per message (matches user's known preference for one-at-a-time dialogue). Never reveal running scores during this phase.

- [ ] **Step 5: Phase 4 — Bonus stretch question**

Instructions: if `bonus_question`, ask one question at `bar_level` + `bonus_offset` (systemic/organisational framing). Score separately; missing it does not reduce the core score, nailing it flags "interviewing above the set bar."

- [ ] **Step 6: Phase 5 — Report (terminal + Markdown + HTML)**

Instructions: compute per-dimension scores + overall + plain-English verdict (§5). Print the terminal summary. Then build the Markdown scorecard in the Task 3 shape and the HTML report by substituting the Task 2 tokens (clone `<!--ROW-->` per dimension). Generate the reading list from the actual gaps (not boilerplate). Print the saved paths. Open the HTML in a browser ONLY if `auto_open`.

- [ ] **Step 7: Phase 5b — Artifact safety enforcement (spec §6a, HARD requirements)**

Instructions, in this exact precedence:
  1. Resolve `save_dir`. Default expands `~` and `<repo-name>` to `~/.interview-me/<repo-name>/` (create if missing).
  2. If the resolved `save_dir` is inside the current repo: ensure that directory is in `.gitignore` (append it if absent) AND check whether it is already git-tracked (`git ls-files`); if tracked, warn the user before writing.
  3. If `include_transcript` is false, omit the transcript section. If `include_diff_snippets` is false, do not quote code from the diff in any saved artifact (terminal is exempt).
  4. If `redact_secrets`, scrub obvious secrets/tokens/URLs from all text written to disk.
  5. Never write review artifacts to a tracked path without the warning in (2).

- [ ] **Step 8: Dry-run verification against a real diff**

Make a trivial throwaway change in a scratch git repo, invoke the skill mentally/via a test run following SKILL.md end to end. Confirm: recon reads the diff; exactly `core_questions` core questions + 1 bonus; a probe fires on a deliberately weak answer; the report writes to `~/.interview-me/<repo>/` (NOT the repo); `.gitignore` untouched because save is out-of-repo; HTML did not auto-open.
Expected: all true. Fix SKILL.md wording for any that fail.

- [ ] **Step 9: Commit**

```bash
git add interview-me-senior/SKILL.md
git commit -m "feat: add five-phase interview logic and artifact-safety enforcement"
```
